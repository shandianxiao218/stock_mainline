| 问题 | 结论 |

## 一、必须先回答的阻塞问题

### 1. 当前版本是 demo 还是正式架构雏形？

答复：当前版本是个人本地 demo，但按可演进架构雏形推进。

当前阶段的正式边界：

- 本地 Web + Python API + SQLite + 东方财富本地数据。
- 面向个人复盘、历史研究、Excel 导出。
- 不承诺公网安全、多人协作、生产级调度、正式权限体系。

可产品化演进方向：

- SQLite 迁移 PostgreSQL。
- `X-User-Role` 迁移登录、会话、用户、权限表。
- 本地脚本迁移任务队列。
- 当前散落 API 迁移 OpenAPI 和统一错误码。
- 快照表、模型版本、聚合版本保留为正式架构基础。

验收口径：当前验收按个人本地研究工具执行，不按商业化 SaaS 或团队平台验收。

### 2. 盘中监控是否属于当前路线？

答复：当前版本明确排除盘中实时监控；后续是否做，需要作为 Phase 4 独立立项。

当前不验收：

- 分钟级行情。
- 盘中涨停/炸板实时状态。
- 盘中舆情刷新。
- WebSocket 或长轮询实时看板。

后续若进入 Phase 4，必须重新定义：

- 实时数据源能力。
- 任务调度频率。
- 延迟指标。
- 前端实时更新机制。
- 盘中评分和收盘评分之间的口径差异。

### 3. 模型输出是否允许被理解为交易信号？

答复：不允许。所有输出只能表达研究排序、风险提示和验证条件，不能表达买卖建议。

强制 UI/API 规则：

- 首页、详情页、复盘报告、Excel 导出必须展示“研究辅助，不构成投资建议”。
- 低置信度时弱化榜单确定性表达，显示“参考价值有限”。
- 次日验证点只能写条件，不写买入、卖出、加仓、减仓。
- 持仓风险只提示暴露方向和风险源，不给交易动作。
- API 增加或保留 `confidence`、`risk_penalty`、`data_quality`/`source_type` 信息，避免只输出单一排名。

后续动作：补齐前端每个模块的数据来源和免责声明展示。

### 4. “真实数据驱动”的边界在哪里？

答复：当前是“部分真实数据驱动”，不是全因子真实数据。

| 模块 | 当前数据状态 | 口径 |
| --- | --- | --- |
| 日线行情 | 东方财富本地二进制，经 C 导入 | 真实本地数据 |
| 股票名称 | 东方财富本地文件解析 | 真实本地数据 |
| 东方财富板块 | 东方财富本地板块成分 | 真实当前快照，历史版本不足 |
| AKShare 热度 | 已接入东财热度排行 | 外部接口数据，但不是完整舆情 |
| 龙虎榜 | 已接入 AKShare 龙虎榜 | 外部接口数据 |
| 舆情因子 | 行情代理 + 热度排行增强 | 代理/近似，不等同新闻和社区全量舆情 |
| 催化事件 | 人工录入 S/A/B/C，已入评分 | 人工数据 |
| 涨停/触板/炸板 | 日线高收低近似 | 近似数据 |
| 资金流 | 未接入真实资金流 | 暂用成交额/涨跌幅代理 |
| 多源板块 | 东方财富为主 | 申万/中信/同花顺/通达信未接 |

验收口径：评分解释必须标注数据来源类型：真实、近似、代理、人工、缺失。

## 二、数据与回测问题

### 5. 如何保证回测没有未来函数？

答复：当前只能保证行情读取路径不使用评分日之后的数据，不能完全保证板块成分和主线定义无未来函数。

已实现：

- 5 年日线覆盖检查。
- 回测结果返回 `fallback_count`、`gap_dates`。
- 无未来函数风险检查能识别板块成分历史版本不足。
- 自动聚合结果已落库到 `local_auto_theme_cluster`。

仍缺：

- 东方财富板块成分只有当前快照，不能证明历史任意日期成分真实。
- 回测时还需要强制使用当时有效的自动聚合版本。
- 人工修正历史生效规则需要更细。

验收口径：当前只能叫“日线行情无未来读取约束回测”，不能叫“完全无未来函数回测”。

### 6. 板块成分历史来源如何验证？

答复：当前无法验证完整历史成分，必须降级提示，不能用当前成分静默回填历史。

策略：

- `em_sector_constituent_history.as_of_date` 有多版本时，按回测日期选择不晚于当日的版本。
- 只有当前快照时，回测输出成分版本风险。
- 不静默把当前成分当成历史真实成分。
- 后续通过定期保存东方财富板块成分快照逐步累积历史。

### 7. 日线近似涨停/触板/炸板是否误导短线情绪因子？

答复：会有误差，必须标注近似信号。

当前口径：

- `touched_limit` 用最高价是否触及涨停价近似。
- `sealed_limit` 用收盘价是否接近涨停价近似。
- `limit_break` 用触板但未封板近似。

失真场景：

- ST、新股、北交所、临停、复牌、价格笼子。
- 盘中多次开板回封。
- 不同涨跌幅制度切换。
- 盘口封单质量无法由日线判断。

后续动作：风险明细和因子解释标注“日线近似”，接入更细粒度数据前不把封板质量表达为精确事实。

### 8. 没有资金流数据时，资金同步性如何落地？

答复：当前暂用成交额、成交额占比、涨跌幅同步性代理，不等同真实资金流。

验收口径：

- “资金同步性”在当前版本应改名为“成交/价格同步性代理”。
- 资金流净流入、ETF 流入、北向资金等未接入前，不在 UI 中表达为真实资金流。

### 9. 交易日历是否必须独立建表？

答复：正式版本必须有独立交易日历；当前 demo 用行情日期推导，只适合本地研究。

风险：

- 数据缺失日可能被误判为非交易日。
- 临时休市、半日市、节假日无法表达。
- 日期回退原因不透明。

后续动作：增加 `trading_calendar`，并在 API 返回 `requested_date`、`resolved_trade_date`、`is_fallback`、`fallback_reason`。

## 三、模型与评分问题

### 10. 权重来自经验还是回测标定？

答复：当前默认权重来自 SRS 经验设定，不是严格样本外标定。

后续必须增加：

- 基准模型对照：涨幅排序、成交额排序、涨停数排序、等权因子排序。
- 样本内/样本外分割。
- 分行情环境测试。
- 权重实验版本对比。

### 11. 风险扣分 0-20 是否匹配？

答复：0-20 是当前经验上限，已满足短期可解释性，但没有完成尺度标定。

后续应验证：

- 极端退潮行情中 20 分是否足够。
- 是否需要动态风险上限。
- 是否需要低置信度时额外折扣主线分。
- 高热度高风险主线是否仍被错误排前。

### 12. 核心股识别是否过粗？

答复：是。当前核心股主要按成交额、涨幅、成分贡献近似，尚未达到 SRS 的龙头/中军/弹性/趋势/后排分类要求。

后续核心股模型：

- 龙头股：连板高度、涨幅、辨识度。
- 中军股：成交额、市值、指数权重。
- 弹性股：波动率、小市值。
- 趋势股：均线趋势、连续相对强度。
- 后排补涨：低位扩散、非核心成交变化。

### 13. 主线阶段是否有状态机？

答复：当前是规则标签，不是完整状态机。

后续状态机需定义：

- 启动、加速、高潮、分歧、退潮、修复的进入条件。
- 状态迁移限制。
- 是否允许复合状态。
- 状态是否影响风险扣分和主线分。

当前验收：只验收状态标签，不验收完整阶段状态机。

### 14. 舆情代理是否重复计分？

答复：存在重复计分风险。

当前处理口径：

- 接入东财热度排行后，舆情因子比纯成交代理更好，但仍不完整。
- 未接入真实社区、搜索、新闻提及前，舆情权重不应继续上调。
- 评分解释必须说明舆情数据来源。

### 15. 催化事件如何进入评分？

答复：已按 S/A/B/C 等级和时间衰减进入评分，但仍依赖人工录入。

当前口径：

- S/A/B/C 等级映射基础分。
- 按时间衰减影响热度和延续性。
- 催化解释显示来源和贡献。

后续需补：

- 新闻自动分类。
- 来源可信度。
- 多事件叠加上限。
- 人工修正审计。

### 16. 因子动态权重是否应该上线？

答复：当前只应作为研究增强和小幅修正，不应作为激进自适应模型。

约束：

- 单日变化限制。
- 5 日和 20 日双窗口一致才调整。
- 最大上下限约束。
- 样本不足时不调整。
- 回测必须记录参数版本，防止污染。

## 四、主线聚合与版本管理问题

### 17. 自动聚合结果未版本化时如何复现？

答复：这是高优先级问题。当前代码已新增 `local_auto_theme_cluster` 保存自动聚合版本，但回测全链路仍需强化。

仍需完成：

- 回测按交易日读取当时自动聚合版本。
- 快照与聚合版本绑定。
- 人工覆盖版本与自动版本差异可追溯。

### 18. 人工主线和自动聚合冲突时谁优先？

答复：人工映射优先，但必须版本化、审计化。

规则：

- 人工映射默认自生效日起向后生效。
- 若要修改历史，必须显式指定历史修正原因。
- 修改、合并、拆分必须写入 `local_theme_version` 和审计日志。
- 回测使用当时有效定义，不默认使用最新人工定义。

### 19. 主线合并/拆分阈值怎么设？

答复：当前是轻量规则，未完成正式阈值标定。

后续阈值需要包含：

- 成分股 Jaccard 重叠。
- 核心股共现。
- 涨停股共现。
- 价格和成交同步性。
- 舆情关键词相似度。
- 人工白名单/黑名单。

AI硬件和国产算力这类高度相关但交易语义不同的方向，需要允许“弱关联但不合并”。

## 五、性能与架构问题

### 20. 100ms 口径是什么？

答复：当前 100ms 指本机服务端快照 API 读取耗时。

不包含：

- 浏览器完整渲染。
- 首次静态资源加载。
- 网络延迟。
- 冷启动快照构建。

当前验证：5 年数据量下，榜单和矩阵快照读取低于 100ms。

### 21. 快照如何失效和重建？

答复：当前有快照构建和状态查询，但缺完整依赖图。

应建立失效规则：

| 变更 | 需重建 |
| --- | --- |
| 行情/板块导入 | 榜单、矩阵、详情、风险、置信度、因子、回测日度快照 |
| 模型参数变化 | 全部评分相关快照 |
| 主线映射变化 | 榜单、矩阵、详情、回测快照、聚合版本 |
| 催化事件变化 | 对应日期及衰减窗口快照 |
| AKShare 热度/龙虎榜变化 | 对应日期评分和详情快照 |

### 22. 冷启动慢是否可接受？

答复：个人本地 demo 可接受首次导入后等待快照构建；正式产品不可接受无进度的等待。

后续需要：

- 快照构建进度。
- 失败重试。
- 部分可用。
- 后台任务队列。
- 增量快照。

### 23. SQLite 是否能承载后续多人、多任务？

答复：不能作为正式多人平台长期方案。

SQLite 适合：

- 单机本地使用。
- 个人研究。
- 只读快照查询。

迁移触发条件：

- 多人同时使用。
- 公网/API 服务。
- 多任务并发写入。
- 更严格权限和审计。
- 更大规模历史数据和实验版本。

目标：迁移 PostgreSQL，并保持 SQLite schema 与正式 DDL 的字段语义一致。

### 27. 前端是否展示数据可信度标签？

## 七、安全、权限、审计问题

### 28. `X-User-Role` 是否正式安全方案？

答复：不是。只允许本地可信环境使用。

必须在文档中明确：

- 不得暴露到公网。
- 不得暴露到不可信局域网。
- 请求头角色不能作为正式鉴权。
- 商业化前必须替换正式登录和会话。

### 29. 审计日志是否防篡改？

答复：当前不防篡改，只是本地操作记录。

正式版本需要：

- 独立审计库。
- 追加写。
- 哈希链或外部日志系统。
- 用户身份绑定。
- 管理员也不能静默修改。

## 八、测试与验收问题

### 30. 验收是否区分功能和模型有效？

答复：必须区分。

验收层级：

- 功能验收：接口、页面、导出、权限、任务。
- 数据验收：覆盖、缺失、来源、历史版本。
- 性能验收：快照 API、冷构建、回测耗时。
- 模型验收：Rank IC、胜率、超额、分组回测、样本外。
- 解释验收：分数拆解、风险原因、数据来源标签。

### 31. C 导入器如何防回归？

答复：需要固定样本和校验脚本。

验收项：

- 固定小样本二进制。
- 行数校验。
- 股票名称校验。
- 日期区间校验。
- 板块成分数量校验。
- 格式变化时失败要明确报错。

### 32. 前端关键路径是否自动化测试？

答复：需要。当前已有前端自动化测试基础，但仍需扩展覆盖。

最低覆盖：

- 日期切换。
- 榜单加载。
- 详情加载。
- 矩阵加载。
- 回测提交和轮询。
- Excel 导出。
- 权限错误。
- 错误面板展示。

## 重点 5 问的正式答复

1. 自动聚合结果已开始版本化，但历史回测必须进一步绑定当日聚合版本，否则只能算“行情日线无未来读取”，不能算完整可复现。
2. UI/API 必须显式标注数据可信度；当前这是高优先级产品缺口。
3. 100ms 只指服务端快照读取；快照失效和重建依赖图仍需补齐。
4. 盘中监控当前明确排除；后续 Phase 4 另行立项，不进入当前验收。
5. SQLite + `X-User-Role` 只限本地 demo；产品化迁移点是 PostgreSQL、正式认证、任务队列、OpenAPI、不可篡改审计。
